# Steering: Configurable Pipeline + Loop Plan Compilation

The spec has been significantly updated. The hardcoded `plan-build-review` cycle is replaced by a **configurable agent pipeline** compiled into `loop-plan.json`. Read the updated SPEC.md sections carefully before implementing.

## What Changed in the Spec

### 1. `loop-plan.json` — the runtime-to-shell bridge

The aloop runtime (TS/Bun) compiles pipeline YAML config into a simple `loop-plan.json` file. `loop.sh` reads this file each iteration — no YAML parsing, no transition logic in shell.

Each entry is a **fully-resolved instruction**:
```json
{
  "cycle": [
    {"agent": "plan",  "prompt": "PROMPT_plan.md", "provider": "claude", "model": "claude-opus-4-6", "reasoning": "high"},
    {"agent": "build", "prompt": "PROMPT_build.md", "provider": "opencode", "model": "openrouter/openai/gpt-5.1", "reasoning": "medium"}
  ],
  "cyclePosition": 0,
  "iteration": 1,
  "version": 1
}
```

- `cycle` is a **short repeating pattern** (5-7 entries). Shell wraps with `cyclePosition % length`.
- `cyclePosition` and `iteration` are in the file — shared state between runtime and shell.
- Shell updates `cyclePosition`/`iteration` after each turn. Runtime rewrites the file on mutations (steering, failure recovery).
- Shell re-reads the file every iteration to pick up runtime mutations.
- `version` increments on runtime rewrites — shell logs when it detects a plan change.

### 2. No more hardcoded phase cycle in loop scripts

`loop.sh` and `loop.ps1` must NOT contain hardcoded phase sequences like `plan → build × 3 → review`. Instead:
- Read `loop-plan.json` each iteration
- Pick agent at `cyclePosition % cycle.length`
- Use `provider`, `model`, `prompt`, `reasoning` from the entry
- Invoke the provider, then update `cyclePosition` and `iteration` in the file

### 3. Pipeline mutation flows through `loop-plan.json`

All mutations (steering, failure recovery, agent injection) work by the runtime **rewriting** `loop-plan.json`:
- Steering says "add security-audit after build" → runtime recompiles cycle array, writes new plan
- Build fails 3x → runtime applies `onFailure` rules, adjusts `cyclePosition` or injects debugger agent
- Loop picks up changes on next iteration (re-reads file)

### 4. Transition rules stay in runtime, not shell

`onFailure: goto build`, escalation ladders, `repeat` counts — all resolved by the runtime when it observes failures via `status.json`. The runtime rewrites `loop-plan.json` accordingly. Shell never evaluates transition logic.

## Implementation Priority

For the TODO items `[pipeline][medium]`:

1. **First**: Create the `loop-plan.json` compiler in the runtime — takes pipeline YAML config (or defaults), resolves providers/models/reasoning, outputs the JSON file. This is the foundation.

2. **Second**: Update `loop.sh` to read `loop-plan.json` instead of using hardcoded phase calculation. The shell code becomes simpler, not more complex.

3. **Third**: Runtime mutation — the host monitor watches `status.json`, applies transition rules, rewrites `loop-plan.json` when needed.

4. **Fourth**: Guard agent and escalation ladder — these are just pipeline configs that the compiler and mutator handle.

## Key Constraint

The pipeline config (`.aloop/pipeline.yml` or defaults) is the **source of truth** for what agents run. `loop-plan.json` is a **compiled artifact** — never hand-edit it, always regenerate from config. The relationship is like TypeScript → JavaScript: you edit the source, the compiler produces the runtime artifact.
