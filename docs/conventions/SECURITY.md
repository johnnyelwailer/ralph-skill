# Security Conventions — Aloop

> Agents read this file to enforce secure-by-default practices, especially around agent sandboxing.

## Agent PATH Sanitization

The inner loop (`loop.sh` / `loop.ps1`) sanitizes the environment before invoking agents:

- **PATH is hardened** to remove dangerous binaries. Only essential tools remain on PATH.
- **`gh` CLI is stripped from agent PATH.** Agents must not call GitHub API directly.
- **`CLAUDECODE` env var is sanitized** to prevent prompt injection via environment.

Why: Agents run untrusted code (generated by LLMs). Limiting their environment reduces blast radius.

## Convention-File Protocol for Side Effects

Agents cannot perform side effects directly. Instead, they write **request files** that the runtime processes:

```
requests/*.json — agent writes a request
runtime reads, validates, executes, writes response
agent reads response in next iteration (baked into prompt by runtime)
```

### Request/Response Envelope Format

```json
{
  "type": "github.pr.create",
  "payload": {
    "title": "feat: add health check",
    "base": "main",
    "head": "feature/health-check"
  },
  "requestId": "req-abc123",
  "timestamp": "2026-03-20T10:00:00Z"
}
```

- Requests are validated by the runtime before execution.
- Unknown request types are rejected.
- The runtime is the only component that touches external APIs.

## Trust Boundaries

| Boundary | Trust Level | Controls |
|----------|------------|----------|
| Agent (inner loop) | Untrusted | Sandboxed PATH, no network API access, file-based communication only |
| Runtime (aloop CLI) | Trusted | Processes requests, calls GitHub API, manages sessions |
| Dashboard | Semi-trusted | Read-only observation + user steering (writes to `STEERING.md`, `queue/`) |
| Orchestrator | Trusted | Dispatches child loops, manages issues/PRs, wave scheduling |

## Secrets

- **Never commit secrets.** No API keys, tokens, passwords, or private keys.
- **Use environment variables** for provider API keys (ANTHROPIC_API_KEY, OPENAI_API_KEY, etc.).
- **`.gitignore` must include:** `.env*`, `*.pem`, `*.key`, `credentials.*`, `secrets.*`, `active.json`
- **If a secret is committed:** rotate it immediately. Removing from git history is not enough.

## Input Validation

- **Validate all request envelopes** from agents before processing. Check type, required fields, payload shape.
- **Validate frontmatter** in prompt files — reject unknown providers, invalid model IDs.
- **Validate JSON state files** (`loop-plan.json`, `status.json`) on load — never trust file contents blindly.
- **Sanitize markdown output** rendered in the dashboard (DOMPurify for any HTML rendering).

## Dependencies

- **Lock dependency versions.** Always commit lockfiles.
- **Minimize dependencies.** The CLI has minimal deps (just `commander`). Keep it that way.
- **Dashboard deps** are more permissive but should still be audited.

## GitHub Enterprise Support

- **Never hardcode `github.com`.** Use `gh` CLI which respects `GH_HOST` and enterprise configs.
- API URLs must be derived from the user's git remote or environment, not assumed.

## Per-Task Sandbox Policy (Orchestrator)

In orchestrator mode, different tasks may need different sandbox levels:
- **Container sandbox** (default) — task runs in isolated Docker container
- **Host OS sandbox** — task needs host access (e.g., Windows-specific builds, hardware tests)
- Sandbox policy is configured per-issue or per-wave in orchestrator config

## Quick Checklist

- [ ] No secrets in code or config
- [ ] `.gitignore` covers sensitive files and runtime state
- [ ] Agent PATH is sanitized before invocation
- [ ] `gh` CLI removed from agent PATH
- [ ] All agent side effects go through request/response protocol
- [ ] Request envelopes validated before execution
- [ ] GitHub Enterprise URLs supported (no hardcoded github.com)
- [ ] Dashboard sanitizes rendered markdown
